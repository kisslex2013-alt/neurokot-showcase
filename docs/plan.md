# NeuroKot Showcase — план разработки (Git-first + Vercel)

> Цель: внутренняя витрина AI‑кейсов/проектов Яндекса с обязательной оценкой эффекта (Impact) и готовыми артефактами для повторного использования (kit).
> 
> Принятые решения:
> - Хранилище кейсов: **Git-first** (кейсы как файлы, review через PR).
> - Деплой: **Vercel**.
> - Авторизация: **можно без неё для MVP**, но в документации и архитектуре учитывать будущую **SSO**.

## 0) Продуктовая гипотеза

**Проблема:** AI‑кейсы/скрипты расползаются по чатам/вики/памяти людей → команды заново изобретают решения, сложно понять эффект.

**Решение:** витрина, где каждый кейс:
- описан в структуре **Problem → Solution → Impact**, 
- имеет **Impact/ROI** (с допущениями + confidence),
- содержит **kit**: prompt pack / runbook / template / ссылки на репо,
- легко добавляется через PR (формой или генератором),
- легко находится через поиск/фильтры.

**Ожидаемый эффект:** рост реюза и снижение времени внедрения (time‑to‑first‑value).

## 1) MVP-объём (то, что нужно успеть и показать)

### Must-have
1. Каталог кейсов + страница кейса.
2. Структурированные данные кейса (schema) + базовая валидация.
3. Фильтры + поиск (минимум: по заголовку/тегам; далее — полнотекст).
4. Impact блок + простой ROI калькулятор.
5. Kit секция (пакет артефактов).
6. "Как добавить кейс" — инструкция.

### Nice-to-have (после MVP)
- AI‑редактор кейса (черновик → структурированный кейс + теги).
- Семантический поиск.
- Интеграции (share‑card, export).
- Auth/SSO.

## 2) Архитектура (простая и расширяемая)

- **Next.js** (Vercel) — UI витрины.
- `cases/` в репозитории:
  - `cases/<slug>/case.json` — структурированные поля
  - `cases/<slug>/README.md` — человеческое описание (опционально)
  - `cases/<slug>/kit/*` — артефакты (prompts, runbook, templates)
- Сборка:
  - на build-time индексируем кейсы в JSON (для быстрого поиска).
- Валидация схемы:
  - JSON Schema + скрипт проверки в CI.

## 3) Данные кейса (минимальная схема v1)

### Поля (v1)
- `id`, `slug`, `title`
- `domain` (например: dev/ml/ops/product/support)
- `tags[]`
- `nominations[]` (creative/optimization/technical/scalable)
- `problem` (коротко)
- `solution` (коротко)
- `impact`:
  - `time_saved_min_per_user_per_day`
  - `users_affected`
  - `confidence` (draft|measured|verified)
  - `notes` (допущения)
- `owner`:
  - `name`, `team`, `contact`
- `kit`:
  - `prompt_pack[]` (paths)
  - `runbook` (path)
  - `repo_url` (optional)
  - `demo_url` (optional)

## 4) План работ (фазы и задачи)

### Этап 1 — Repo bootstrap + schema (0.5–1 день)
- [ ] Создать папки `app/` (Next.js), `cases/`, `docs/`.
- [ ] Добавить JSON Schema `cases/schema.case.v1.json`.
- [ ] Добавить 2–3 примерных кейса (заглушки) для демо.
- [ ] CI: скрипт `npm run validate:cases`.

**DoD:** сборка проходит, schema валидируется, витрина показывает кейсы.

### Этап 2 — UI витрины (1–2 дня)
- [ ] Страница каталога: карточки, фильтры, сортировки.
- [ ] Страница кейса: Problem/Solution/Impact/Kit.
- [ ] Поиск по title/tags (простая реализация).

**DoD:** удобно пользоваться без AI, быстрый load, понятные карточки.

### Этап 3 — Impact/ROI (0.5–1 день)
- [ ] Калькулятор: время→часы/нед→₽ (опционально).
- [ ] Отображение confidence + допущений.

**DoD:** каждый кейс имеет сравнимый Impact.

### Этап 4 — Intake через PR (1 день)
- [ ] `docs/how-to-add-case.md`.
- [ ] Скрипт генерации кейса: `scripts/new-case.mjs` (создаёт папку, template json, kit).

**DoD:** новый кейс добавляется за 2–5 минут.

### Этап 5 — AI‑редактор (опционально, 1–2 дня)
- [ ] Страница "Draft → Case" (ввод текста, вывод структурированного JSON).
- [ ] Автотеги + предложение номинации.

**DoD:** качество описаний выравнивается.

### Этап 6 — Документация для SSO (0.5 дня)
- [ ] `docs/auth-and-sso.md`: варианты (без auth / proxy-auth / OIDC).

**DoD:** в проекте явно описан путь к SSO.
